Projet personnel FastAPI Docker CI/CD Git GitHub Actions Hugging Face Hub Hugging Face Spaces LightGBM Pydantic Pytest Logs JSON Evidently (drift) MLflow (runs) Profiling (cProfile)
  • Endpoints : `GET /health`, `POST /predict`.
  • Modèle servi : pipeline LightGBM sérialisé (joblib) - téléchargé depuis Hugging Face Hub.
  • Déploiement démo : Hugging Face Spaces (Docker) - service accessible publiquement.
  • Cycle MLOps couvert : API stable → tests → CI/CD → logs → monitoring → optimisation des performances.
Ce que le projet démontre (côté prod) :
  • CI/CD : à chaque commit → exécution des tests → build image Docker → déploiement.
  • Tests : `/predict`, validation Pydantic, erreurs 422 (input) vs 500 (système), cohérence des réponses.
  • Logging structuré JSONL : request-level + événements `prediction` / `validation_error` (fichier `logs/api_requests.jsonl`).
  • Détection d’anomalies : analyse statistique des latences + test KS sur distributions.
  • Optimisation perfs : profilage (`cProfile`) pour identifier un goulot côté chargement/initialisation, puis optimisation du chemin de chargement.
Chiffres (extraits du POC) :
  • Métriques ML : AUC ~0.77 selon le comparatif multi-modèles (CV).
  • Latence /predict (42 requêtes) : mean 262.70 ms ; median 0.92 ms ; max 2599.65 ms.
  • Note latence : 42 lignes /predict dans les logs, 40 `request_id` uniques (doublons présents).
  • Drift (Evidently, K‑S) : signal de dérive sur `probability_default` — p-value = 0.015873… (< 0.05), `dataset_drift = true`.
    Posture prod : indicateur à investiguer (échantillon limité, drift sur scores uniquement) → analyse cause + suivi perf + éventuel retraining/recalibration.

Puls-Events RAG

Système Retrieval-Augmented Generation (RAG) fonctionnel basé sur LangChain et le modèle Mistral, soutenu par une base vectorielle FAISS — OpenAgenda : ingestion → index FAISS → API FastAPI → garde-fous anti-hallucination + évaluation.

FastAPI Docker Docker Compose FAISS LangChain sentence-transformers Mistral API
  • Contexte (scénario OC) : mission freelance pour Puls-Events — chatbot événements (POC).
  • Données : OpenAgenda (périmètre Paris-Saclay) (POC).
  • Index : FAISS persistée ; embeddings `all-MiniLM-L6-v2` (local) ; chunking 800, overlap 120.
  • API : `POST /ask` (question → réponse sourcée) ; `POST /rebuild` (reconstruction index).
  • Qualité : réponses sourcées + évaluation reproductible (gold set).
Ce que le projet démontre (POC) :
  • Architecture RAG de bout en bout : pipeline offline (collecte/nettoyage/chunking/index) + exécution online (retrieval FAISS → génération LLM).
  • Traçabilité : réponse construite à partir des documents récupérés (sources), pour limiter l’hallucination.
  • Exposition : API FastAPI testable via Swagger ; exécution locale via Docker Compose.
  • Limites assumées : dépendance à un LLM externe (Mistral), latence variable, périmètre géographique restreint.
Chiffres (extraits du POC) :
  • Volumétrie : ~3800 événements bruts ; ~661 indexés.
  • Gold set (10 Q) : 5 correct, 5 partial, 0 incorrect ; precision=0.6183 ; recall=1.0 ; f1=0.69.

Attrition RH — TechNova Partners

Projet de classification supervisée en contexte RH : explorer 3 sources de données (SIRH, évaluations annuelles, sondage), prédire une probabilité de départ et interpréter les facteurs clés.

Classification supervisée EDA Régression logistique Imbalance Validation croisée Seuil de décision One‑Hot Encoding Analyse d’erreurs
  • Volumétrie (extraits) : 1470 employés ; 237 démissionnés (~16%).
  • Baseline : Dummy Classifier accuracy ≈ 84% mais ne détecte pas la classe minoritaire.
  • Modèle retenu (extrait) : régression logistique ; recall(« départ ») ≈ 0.68.
  • Compromis métier : ajustement du seuil (precision/recall) pour « mieux alerter que rater ».
Ce que le projet démontre (POC) :
  • Fusion multi-sources RH : structurer un dataset unique à partir de 3 fichiers (SIRH, évaluations annuelles, sondage) avec recouvrement via identifiants.
  • Préparation : nettoyage, suppression de colonnes non pertinentes/constantes, encodage catégoriel (One‑Hot), gestion de corrélations fortes.
  • Évaluation adaptée au déséquilibre : aller au-delà de l’accuracy (precision/recall, matrice de confusion) et comparer des modèles via validation croisée.
  • Décision métier : piloter le compromis faux positifs / faux négatifs avec un seuil de décision ajustable.
  • Interprétation : lecture des coefficients (logistique) + analyse FN/FP pour comprendre les limites du modèle.
Chiffres (extraits) :
  • Données : 3 sources (SIRH, évaluations annuelles, sondage).
  • Classe minoritaire : 237 démissionnés (~16%).
  • Après préparation (extrait) : 1233 employés restants.
  • Dummy : accuracy ≈ 84% mais recall sur « départ » ≈ 0.
  • Régression logistique : recall(« départ ») ≈ 0.68 (seuil 0.50, extrait).
  • Effet du seuil (extraits) : 0.20 → recall=0.91 / precision=0.22 ; 0.90 → precision=1.00 / recall=0.17.

Semi‑supervisé (Computer Vision) — BrainScanAI

Démo technique de classification d’images IRM (tumeur vs normal) avec peu de labels : extraction d’embeddings, clustering, pseudo‑labels et comparaison supervisé vs semi‑supervisé, avec garde‑fous anti fuite de données.

Computer Vision Semi‑supervisé Embeddings (ResNet50) Clustering (KMeans/DBSCAN) Pseudo‑labels ARI F1 macro Data leakage QC données
  • Données : 1506 images ; 100 labellisées (50/50) ; 1406 non labellisées.
  • Prétraitement : gris → RGB, resize 224×224, normalisation ImageNet (compatibilité backbones pré‑entraînés).
  • Pipeline : embeddings → clustering → mapping cluster→classe (train only) → pseudo‑labels filtrés → entraînement + comparaison équitable (test strict).
Ce que le projet démontre :
  • Structuration d’un pipeline semi‑supervisé avec séparation train/val/test et interdiction d’utiliser le test pour guider les choix.
  • Feature extraction via modèle pré‑entraîné (embeddings) pour travailler sur des représentations plutôt que sur les pixels.
  • Contrôle qualité (outliers/quasi‑doublons) et garde‑fous pour préserver le jeu labellisé.
  • Comparaison méthodique : baseline supervisée vs semi‑supervisé + robustesse via multi‑seeds.
  • Projection “passage à l’échelle” : pipeline batch relançable + traçabilité (artefacts) + monitoring simple.
Chiffres (extraits) :
  • Embeddings : 2048 dimensions/image ; matrice 1506×2048.
  • Pseudo‑labels : ~1406 générés puis filtrage (≈ un quart conservé, autour de 350 images).
  • Test final (extraits) : supervisé ~95% accuracy / F1 macro ~0.95 ; semi‑supervisé ~90% / ~0.90, avec stabilité accrue (variance plus faible) en multi‑seeds.
  • Budgets mentionnés : 300 € (dataset) ; 5 000 € pour 4 M d’images à labelliser.

Reinforcement Learning LunarLander‑v3 PPO DQN Reward API Swagger (/docs) Dashboard
