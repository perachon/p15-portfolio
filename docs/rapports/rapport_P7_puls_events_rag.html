Rapport de conduite de projet AI Engineering — Puls-Events RAG OpenAgenda

1. Contexte et analyse des besoins

1.1 Présentation (organisation et / ou contexte)

  • Contexte : assistant conversationnel (RAG) pour répondre à des questions sur des événements culturels (récents < 1 an et à venir) sur un périmètre géographique (Paris‑Saclay).
  • Type d’organisation (scénario OC) : mission freelance pour Puls-Events, entreprise tech qui développe une plateforme de recommandations culturelles.
  • Enjeux : améliorer la découverte d’événements pertinents, réduire le temps de recherche, et démontrer la faisabilité d’un chatbot basé sur OpenAgenda.
  • Niveau de maturité IA / MLOps : POC technique — pipeline data + index FAISS + API + Docker ; pas de déploiement “prod”/SLO/alerting.

1.2 Besoin métier

  • Besoin : répondre en langage naturel et recommander un top événements avec date/lieu/ville + lien/source.
  • Parties prenantes (scénario OC) : Jérémy (responsable technique) + équipes Produit et Marketing (cibles de la démo).
  • Qualité : éviter l’hallucination ; réponses traçables/sourcées via les documents récupérés.
  • Fraîcheur : événements récents (< 1 an) et à venir → reconstruction de l’index si les données évoluent.
  • Attendus (mission) : RAG LangChain + Mistral + FAISS + scripts de (re)construction d’index + API REST + jeu de test annoté + tests unitaires + rapport technique.

2. Solution (POC)

  • Données : OpenAgenda (agenda universite-paris-saclay).
  • Volumétrie : ~3800 événements bruts ; dataset indexé ~661 (data/processed/events_index_ready.jsonl).
  • Pipeline : ingestion → nettoyage → indexation FAISS persistée (data/index/faiss_events/).
  • Architecture : séparation offline (collecte/nettoyage/chunking/embeddings/index) vs online (retrieval FAISS → génération LLM à partir des documents).
  • RAG : LangChain + LLM Mistral (mistral-small-latest).
  • Embeddings : sentence-transformers/all-MiniLM-L6-v2 (local).
  • Chunking : 800 caractères ; overlap 120.
  • API : FastAPI (/ask, /rebuild).
  • Docker : démo locale ; exécution facilitée via Docker Compose (POC).

Contraintes & limites (POC)

  • Dépendance à un LLM externe : connexion Internet + service tiers (Mistral) ; coût par requête à estimer.
  • Qualité liée à la couverture/qualité des données indexées.
  • Temps de réponse variable selon la requête.
  • Périmètre géographique volontairement restreint pour garder des recommandations cohérentes.
  • Sécurité : secrets via variables d’environnement ; auth API non implémentée (POC).

Évaluation (extraits)

  • Gold set (10 questions) : 5 correct, 5 partial, 0 incorrect.
  • Moyennes : precision=0.6183 ; recall=1.0 ; f1=0.69.
