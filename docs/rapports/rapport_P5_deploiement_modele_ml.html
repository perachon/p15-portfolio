Rapport de conduite de projet AI Engineering — P5 (Déploiement d’un modèle ML / Futurisys)

1. Contexte et analyse des besoins

  • Contexte (scénario) : mission freelance pour Futurisys (demande du directeur technique Aurélien) afin de rendre un modèle ML opérationnel et accessible via une API.
  • Objectif : solution de déploiement prête pour un usage “type production” (POC) incluant versioning, tests, CI/CD et traçabilité.

2. Périmètre et livrables (extraits)

  • Dépôt Git structuré : code, requirements.txt, historique de commits, branches, tags/releases.
  • API FastAPI : endpoints documentés (Swagger/OpenAPI).
  • Tests : suite Pytest + rapport de couverture (pytest-cov).
  • Base PostgreSQL : tables + script de création (SQL/Python) + exemples d’entrées.
  • Pipeline CI/CD : automatisation tests & déploiement + gestion des secrets.

3. Solution (POC) et architecture

3.1 Stack (extrait)

  • Python 3.9
  • API : FastAPI
  • Validation : Pydantic (validation stricte)
  • DB : PostgreSQL + SQLAlchemy (avec fallback SQLite selon environnement)
  • CI/CD : GitHub Actions
  • Déploiement : Hugging Face Spaces (SDK Docker)
  • Tests : Pytest + Coverage (~85% annoncé)

3.2 Endpoints et documentation (extraits)

  • /docs : Swagger UI
  • GET /health : check API + DB
  • POST /predict : endpoint simple (test CI/CD)
  • POST /p4/predict : endpoint modèle d’attrition (artefacts issus du projet 4)

3.3 Modèle et artefacts (extraits)

  • Artefacts réutilisés : pipeline .joblib + schéma d’entrée (input_schema.json avec ordre des features).
  • Objectif : séparation code / modèle et cohérence entre requêtes et modèle servi.

3.4 Traçabilité et base de données (extraits)

  • Tables : employees (dataset RH) et prediction_logs (traçabilité des requêtes et prédictions).
  • Objectif : reproductibilité et auditabilité des prédictions.

4. CI/CD, versioning et qualité

4.1 Workflow Git (extraits)

  • Trunk-based development : main + branches courtes feat/...
  • Commits atomiques
  • PR avec CI obligatoire
  • Tags/Releases pour marquer les versions (exemples : v0.3.0, v0.4.0, v0.5.0)

4.2 CI/CD (extraits)

  • Installe les dépendances → exécute Pytest → calcule la couverture.
  • Si la CI est verte : déploiement sur Hugging Face.
  • Gestion des secrets : HF_TOKEN en GitHub Secrets.

5. Tests & couverture (extraits)

  • Unitaires : validation Pydantic (champs manquants/mauvais type → 422), tests du loader d’artefacts (fichiers manquants, clé feature_order absente…).
  • Fonctionnels : tests endpoints (200 / 400 / 422 / 503) avec mocks ciblés.
  • Intégration : API ↔ DB (un appel /p4/predict → +1 ligne dans prediction_logs).
  • Couverture : ~85% (extrait).

6. Limites & next steps (extraits)

  • Ajouter des tests de performance (latence max) et des tests de non-régression du modèle.
  • Évolutions possibles : endpoint /logs/last, export des logs vers dashboard, migration DB managée (ex. Supabase).
